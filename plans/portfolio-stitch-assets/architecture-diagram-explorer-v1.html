Variation A
Minimal overview (6 nodes)
Shows the system at the highest level. Browser, Worker, D1, and three external services. The Vendor Gateway is called out as a label on the connection, not a separate node. Tests whether a simpler diagram reads better in context.
Architecture
Four layers between a bug and your wallet
Planner V1 system components. Dashed outlines are external services.

The Vendor Gateway is the single choke point for every paid API call. Four layers fire in sequence: killswitch (boolean, instant), global budget cap (atomic counter), per-user budget cap (20 SMS/day), and idempotency (UNIQUE constraint INSERT before the vendor call). A bug in the scheduler can loop forever, but it cannot send two reminders for the same task on the same day.

Notes on this variation:
  • 6 top-level nodes plus 4 internal modules. Reads quickly.
  • Vendor Gateway appears as a label, not a node. Might underplay the most interesting architectural decision.
  • External services are visually distinct (dashed, dimmer text) without introducing a new colour.
  • Cylinder effect on D1 is subtle but conventional for databases.

  • More vertical space required. Might feel tall in a mobile viewport.
  • The numbered protection layers (1. killswitch, 2. budget, etc.) inside the Gateway node directly support the prose paragraph below.
Standards
Architecture diagram colour rules
The colour decisions applied across all three variations, and why.
Royal Tonal only. No Violet Signal.
  • Violet Signal is the "thinking/writing" register. Architecture diagrams are "work/craft" content in the Projects collection. Mixing registers in a single diagram breaks the semantic colour system.
  • If Violet appeared on the Claude Haiku node (because it is AI), then Violet would mean "AI" in diagrams but "writing" everywhere else. The exception would erode the rule.
Three visual tiers within Royal Tonal:
  • Owned components (your code): royal-3 fill + royal-7 stroke. Solid lines. These are the things you built.
  • Highlighted subsystem (the interesting architectural decision): royal-4 fill + royal-8 stroke at 2px. Max one per diagram. The prose paragraph immediately following the diagram explains why this node is highlighted.
  • External services (vendor APIs): no fill + royal-6 dashed stroke + ink-dim labels. Visually recessive. These are dependencies, not decisions.
Arrow hierarchy:
  • Internal data flow: royal-7 at 1.5px (solid)
  • Highlighted flow (path to the interesting subsystem): royal-8 at 2px (solid). Max 1-2 per diagram.
  • External calls: royal-6 at 1.5px (dashed). Matches the external node stroke.
Labels:
  • Node name: Geist 12-13px weight 500, ink colour
  • Node sub-label: Geist Mono 9-10px, ink-dim
  • Arrow label: Geist Mono 9-10px, ink-dim or royal-11
  • Group/band label: Geist 10px weight 500, ink-faint, letter-spacing 0.12-0.18em
Consistency with other diagram types:
  • All diagram types share: 1.5px base stroke, 8px radius, Geist labels, Royal Tonal palette.
  • Architecture diagrams add: grouping boxes, filled/unfilled node distinction, trust-band labels.
  • Flowcharts and decision trees do NOT use grouping boxes or the filled/unfilled distinction.
  • The royal-8 accent highlight is available in all diagram types for the "interesting" node, not just architecture diagrams.
