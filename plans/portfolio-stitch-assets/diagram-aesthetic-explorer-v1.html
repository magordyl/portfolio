Diagram aesthetic explorer

Five candidate aesthetics for the portfolio's diagram layer. The same sample flow rendered in each. Pick one (or one per archetype) to lock the rule in Part 3 of writing-style.md and §Diagrams of portfolio-imagery-standards.md.

How to compare

  • Line quality. Wobbly vs. precise. Wobble signals "drew this to think." Precision signals "drew this to ship."
  • Label weight. Should labels feel handwritten, typeset, or console-logged?
  • Background. Solid flat vs. ruled grid vs. nothing. Grid adds a technical register; nothing lets the shape breathe.
  • Fill strategy. Filled nodes read as containers; outlined nodes read as regions or references.
  • Where it breaks. Every aesthetic has an archetype where it doesn't work. Name the break before locking.

§1The sample flow

A linear process with five nodes drawn from the-weekly case study: how a user goes from browsing recipes to a consolidated trolley. Same nodes, same positions, across every aesthetic below, so the only variable is visual treatment.

A1Hand-drawn sketch

Mermaid look: handDrawn + RoughJS wobble. The aesthetic originally recommended in the standards doc. Cutler's default. Simulated here with an SVG feTurbulence displacement filter.

Hand-drawn sketch
Drew to think
How it readsWobbly strokes, filled containers, labels in Geist sans. Invites the reader into the reasoning.
Signal"I drew this to think." Generous, unpolished, conversational. Good for Process and Lessons sections.
BreakAt small sizes (mobile, thumbnail, social preview), wobble obscures labels. Filter cost on complex diagrams can flicker.
Archetype fitStrong: flowcharts, causal loops. Medium: networks. Weak: 2×2 (grid lines lose grid-ness when wobbled).

A2Clean vector Royal Tonal

Precise strokes, sharp-but-rounded corners, Royal Tonal monochrome. Mermaid default theme without the hand-drawn look. "Drew to ship."

Clean vector Royal Tonal
Drew to ship
How it readsTechnical, calm, confident. Nothing distracting between label and line.
Signal"This is the final shape." Best for Outcome and Architecture sections where conclusions are the story.
BreakRisks looking like marketing-site flatware if fills or shadows creep in. Needs discipline to stay crisp.
Archetype fitStrong: everything, including 2×2 grids and networks. The default safe choice if you pick one aesthetic for all four archetypes.

A3Typographic (ASCII-inspired)

No drawn shapes. Box-drawing characters in Geist Mono, Royal Tonal 8 for emphasis, Royal Tonal 6 for muted. "Console-log aesthetic." The most distinctively developer-native option.

How it readsA rendered terminal dump. The medium signals the subject (code, engineering, build-log).
Signal"Drew this the way I draw everything else: in text." Matches the "learning to ship with Claude Code" narrative directly.
BreakHard to show decisions and branches. 2×2 and causal loops are almost impossible in pure ASCII without looking like a maze.
Archetype fitStrong: linear flowcharts. Weak: 2×2. Very weak: causal loops, networks. Would need a paired aesthetic for other archetypes.

A4Notion-flat

Soft filled nodes, no outlines, muted arrow strokes. The "doc diagram" aesthetic. Less Cutler, more Notion or Linear docs page.

Notion-flat
Drew at rest
How it readsQuiet, polite, doc-page calm. Softer than the clean vector option.
Signal"I'm here to inform, not to persuade." Closer to docs than to case-study prose.
BreakReads as generic. No editorial voice; looks like every SaaS onboarding flow. Risks the "good design obscures poor logic" Kao problem.
Archetype fitStrong: flowcharts. Medium: 2×2. Weak: causal loops (no stroke weight to carry loop direction).

A5Blueprint / architectural

Thin royal-8 strokes on royal-12 background, ruled grid, mono all-caps labels, dimension lines and node numbers. The "engineering drawing" register. No fills.

Blueprint
Drew in pen
How it readsSchematic, technical, intentional. Every element is labelled; every dimension is annotated.
Signal"This was engineered." Reads as an architectural spec, not a napkin sketch. Rewards close looking.
BreakCan read as cold or over-formal for Lessons sections. The grid background clashes with dark prose pages if not contained.
Archetype fitStrong: architecture diagrams, systems. Medium: flowcharts (a bit formal for them). Weak: causal loops (loops fight the ruled grid).

Decision panel

Options in order of increasing editorial voice. None of these are locked yet. The writing-style rule currently reads "pending design explorer preview" and points here.

Option Best at Works for all 4 archetypes? Editorial voice
A1 Sketch Process, Lessons Mostly (2×2 is shaky) High — "drew to think"
A2 Clean vector Architecture, Outcome Yes — the default safe choice Low — "drew to ship"
A3 Typographic Linear flows, build logs No — needs a paired aesthetic High and very specific — "drew with code"
A4 Notion-flat Doc-style info diagrams Mostly Low — "drew at rest"
A5 Blueprint Architecture, systems No — causal loops fight the grid Medium — "drew in pen"

Three ways to lock this:

  1. Single aesthetic for all archetypes. Simplest rule. A2 is the only option that fits this cleanly. Low risk, low voice.
  2. Paired aesthetic: default + accent. Pick one for most diagrams and reserve another for a specific register. Example: A1 as default + A3 for terminal/build sequences only.
  3. Per-archetype mapping. Pick the strongest option for each of the four archetypes. Highest voice, most discipline required. Example: A1 for flowcharts, A2 for 2×2, A1 for causal loops, A5 for architecture diagrams.

My read: option 2 with A1 as default + A3 reserved for sequences that are literally about code or build steps. A1 carries the editorial voice; A3 earns its place by matching the subject. Skip A4 and A5 until there is a specific architecture diagram that demands them.
