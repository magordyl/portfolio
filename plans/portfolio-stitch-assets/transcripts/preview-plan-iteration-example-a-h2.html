Plan: Where these changes land in the existing plan

I've read enough of the portfolio plan to propose concrete changes. Here's what I found and what I'd change — with questions at the end.

Where these changes land in the existing plan

Three chunks are touched:

  • 4a.3 (portfolio-imagery-standards.md) — Chat transcripts section: frame spec is currently locked with text-only sender labels and zero JS / no interactivity. Both improvements contradict this; we need to amend the spec before 4c.1 builds the component.
  • 4a.6 (portfolio-implementation.md) — bookmark workflow: schema and capture script need small additions to support collapsible turns.
  • 4c.1 — <ChatTranscript> component build: changes the render layer.
  • New addition — writing topics workflow, layered on top of the existing portfolio-writing-brainstorm.md and the /bookmark skill.

Improvement 1 — Visual identity for Claude and Dylan

Current frame spec (locked in 4a.3): sender identity is text only — "Dylan" in Royal Tonal 300, "Claude" in Royal Tonal 500, both in Geist Mono caption size.

Proposed change: add a small icon badge left of the sender label line. lucide-react is the workspace icon library; both icons render as static inline SVG in the .astro component, zero JS.

  • Dylan: User (or UserCircle) — Royal Tonal 300, inside a 24px Royal Tonal 900 circle
  • Claude: Sparkles (or Bot) — Royal Tonal 500, inside a 24px Royal Tonal 900 circle
  • Text label stays (a11y, and keeps the "Dylan / Claude" naming explicit for the reader)
  • Colour stays Royal Tonal only — the transcript is still the existing visual register, just with an added glyph

Why this is low-risk: icons are purely additive; no layout changes, no new dependencies, no JS. The only frame spec edit is the sender-label row.

Improvement 2 — Collapsible sections in transcripts

The bigger amendment. The current spec explicitly says *"Zero client-side JS — static render, no interactivity. No expand/collapse on tool calls."* The user ask directly inverts that for long assistant replies.

Proposed change: progressive disclosure via native HTML <details>/<summary> — still zero JS, zero framework, full keyboard a11y, works in static Astro render. Native <details> is HTML semantics, not "interactivity" in the JS/React sense, so the no-JS rule can stay true in spirit.

Schema extension (4a.6.1):

turns: Array<{
  role: 'user' | 'assistant',
  text: string,
  summary?: string,            // NEW — when set, text collapses behind this
  collapsedTools?: string[]
}>

Render rules (4c.1):

  • If turn.summary is set → render <details><summary>{summary}</summary>{text}</details> with "Show full reply ↓" affordance
  • If turn.summary is absent and turn.text exceeds a character threshold (proposed: 800 chars) → auto-collapse, with summary = first sentence or first 140 chars + "…"
  • Short turns → render as-is, no <details> wrapper
  • Tool calls stay always-collapsed (the existing 4a.3 rule stands)

Capture script extension (4a.6.2): when writing the draft, for any assistant turn >800 chars, auto-populate summary with the first sentence. User can edit or clear it during hand-review.

Frame spec amendment (4a.3): change the "Zero client-side JS — static render, no interactivity" line to: *"Zero client-side JS. Progressive disclosure uses native <details>/<summary> only — no React, no framework."* Add a row to the frame-spec table for the summary (Geist italic, Royal Tonal 400) and the expand affordance.

Improvement 3 — Writing topics tracker

What already exists: portfolio/plans/portfolio-writing-brainstorm.md — a freeform markdown doc with tags [seed] → [researched] → [drafting] → [shipped]. Currently one entry (Code-first artefacts). It's manual: no in-flow capture.

What the user wants: a low-friction way to add a seed topic from *any* working directory, usually (but not always) paired with a transcript bookmark, with enough context to jog memory later.

Proposed new sub-chunk 4a.7 — Writing topic capture workflow (sibling to 4a.6):

  1. New /writing-topic skill at workspace level (.claude/skills/writing-topic/SKILL.md in workspace repo — not portfolio repo, because it must work from any project's CWD). Wraps a new scripts/writing-topic.mjs that lives in portfolio repo.
  1. Skill inputs:
  • --slug <kebab> — required
  • --title "..." — required, one-line working title
  • --thesis "..." — optional but encouraged; exact wording of the point
  • --transcript <slug> — optional link to a transcript bookmarked in the same session
  • --context "..." — optional free-text (what was I doing when this hit me)
  1. Storage: I'd recommend changing from the single-file brainstorm doc to a folder of per-topic files at portfolio/plans/writing-topics/<slug>.md, with portfolio/plans/writing-topics/INDEX.md as the scan list. Reasons:
  • One file per topic = safer concurrent appends, cleaner diffs, fits the same pattern as ideas/
  • The existing portfolio-writing-brainstorm.md entry migrates to writing-topics/code-first-artefacts.md during the chunk
  • Adds a new topic = one Write call, not an Edit into a growing file
  • INDEX.md is the 1-line-per-topic scan list (same pattern as ideas/INDEX.md)
  1. Frontmatter per topic file:

``yaml --- slug: code-first-artefacts title: "Build the tool, not the tooling" status: seed captured: 2026-04-12 transcripts: [planner-v1-story-map] # optional list of transcript slugs source_project: planner-app # auto-detected from CWD when captured --- `` Body = thesis + freeform notes + context.

  1. Pairing with /bookmark: when /bookmark runs and the user also wants a writing topic, Claude chains the two calls in-flow: /bookmark --slug X then /writing-topic --slug X --transcript X. Not a merged command — two skills, one prompt, explicit intent.
  1. /session-end integration: extend the existing 4a.6.5 session-end wiring to also list new writing-topics/ entries created this session, with the same "promote / edit / delete" prompt pattern.
  1. First-use gate: before closing 4a, capture one real writing topic via the skill (can use the existing "Code-first artefacts" as the test case by migrating it through the new flow).

Tensions to flag

  1. /bookmark skill location. The current 4a.6.4 plan puts /bookmark inside the portfolio repo at .claude/skills/bookmark/. That makes it unreachable from any other project's CWD (e.g. when you're working in planner-app/ and want to bookmark the moment). If we're adding /writing-topic at workspace level for cross-project use, /bookmark should move with it. The script can still live in portfolio/scripts/ — the workspace skill just invokes it by absolute path.
  1. Locked rule change. The "zero interactivity" rule in 4a.3 was deliberate. Amending it to allow native <details> is a real scope-expansion decision that should be acknowledged in the diary, not slipped in silently.
  1. Character threshold for auto-collapse. 800 chars is a guess. The real number wants to be calibrated against a couple of real assistant turns that match "long plan that takes pages of content" — might be 1200. We should set it and plan to revisit after 4c.1 renders real transcripts.

Questions before I write the final plan

  1. Icon choice for sender badges —

(a) User + Sparkles (stylised, "AI sparkle" is a common visual cue) (b) User + Bot (literal — looks like a robot icon) (c) UserCircle + Sparkles (d) Defer the exact glyph choice to the 4c.1 design explorer — build two variants in case-study-v1.html vs v2.html and pick one there

  1. Icon + text label, or icon only?

(a) Icon + "Dylan" / "Claude" text label (current plan + glyph) (b) Icon only (cleaner, visually stronger, but weaker a11y and first-glance parse) (c) Icon + text on desktop, icon only on mobile

  1. Collapsible mechanism —

(a) Explicit summary field only — author marks what's collapsed during hand-review (b) Auto-collapse only — any turn over the char threshold auto-folds with a generic "Show full reply" (c) Both: auto-fills summary with first sentence when capturing long turns, hand-review can override, render logic doesn't care which path set the value *(my recommendation)*

  1. What does the collapsed "key bits" view show for a long Claude plan? —

(a) First sentence, truncated (b) First N chars (140? 200?) with ellipsis (c) First paragraph (up to first blank line in the turn text) (d) First heading (##, ###) if one exists, otherwise fall back to (a)

  1. Writing topic capture — skill shape —

(a) New /writing-topic skill at workspace level, separate from /bookmark *(my recommendation)* (b) Extend /bookmark with a --topic flag that also writes a topic seed (c) Extend the existing workspace /new-idea skill with a type: writing variant (reuses the ideas/ pattern, but destination is portfolio not workspace — awkward)

  1. Writing topic storage format —

(a) One file per topic under portfolio/plans/writing-topics/<slug>.md + INDEX.md scan list *(my recommendation — matches the ideas/ pattern and the existing brainstorm doc migrates cleanly)* (b) Keep portfolio-writing-brainstorm.md as a single appended file (c) A new Astro content collection writing-ideas with a Zod schema (more rigorous, but adds build-time coupling; overkill for seed notes)

  1. Relocate /bookmark skill to workspace level along with the new /writing-topic skill? —

(a) Yes — both at workspace .claude/skills/, both call scripts living in portfolio/scripts/ *(my recommendation, fixes a latent cross-project gap in 4a.6)* (b) No — keep /bookmark portfolio-local; only add /writing-topic at workspace level (c) Both at portfolio level — means they only work when CWD is portfolio (punts the cross-project problem)
