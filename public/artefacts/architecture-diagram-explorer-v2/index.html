Colour Fixes
Contrast improvements
v1 used royal-3 fill + royal-7 stroke for owned nodes. On a royal-2 background, the fill was barely visible and the stroke was too subtle at small sizes. v2 bumps the palette up.
v1 — too subtle
v2 — improved
Changes:
  • Owned nodes: royal-3 fill + royal-7 stroke → royal-4 fill + royal-8 stroke. The fill now has visible contrast against the background, and the stroke matches the brand blue.
  • Highlighted node: royal-4 fill + royal-8 stroke → royal-5 fill + royal-9 stroke. One step brighter than owned nodes, so it still stands out when everything else also uses royal-8.
  • External nodes: royal-6 stroke → royal-7 stroke. Still dashed, still recessive, but now actually visible at small sizes.
  • External labels: ink-dim → ink (full contrast). At mobile sizes, dimmed labels on dashed-stroke nodes were unreadable. The dashed stroke already signals "external"; the label doesn't need to be dim too.
  • Arrow base colour: royal-7 → royal-8. Arrows need to be visible at 50% scale on mobile.
  • Highlighted arrows: royal-8 → royal-9. Keeps the hierarchy: base arrows are royal-8, highlighted path is royal-9.

The Vendor Gateway is the single choke point for every paid API call. Four layers fire in sequence: killswitch (boolean, instant), global budget cap (atomic counter), per-user budget cap (20 SMS/day), and idempotency (UNIQUE constraint INSERT before the vendor call). A bug in the scheduler can loop forever, but it cannot send two reminders for the same task on the same day.

M1: Horizontal scroll
  • Diagram has a 480px minimum width and scrolls horizontally when the container is narrower.
  • "scroll →" hint appears on narrow viewports.
  • Keeps the same diagram at all breakpoints: one SVG to maintain.
  • Common pattern on technical blogs (Stripe, Vercel, Linear).
  • Downside: part of the diagram is hidden until the user scrolls. First impression may not include the most interesting part.

M2: Full-width bleed
  • Diagram bleeds to container edges (negative margins cancel the card padding).
  • Uses the full 390px viewport width instead of ~234px (after padding).
  • SVG scales to fit: at 390px, the 600-unit viewBox renders labels at ~8px equivalent. Borderline readable.
  • No scrolling required: the full diagram is visible at once.
  • Downside: still small. Labels are tight at this scale. Works for simple diagrams; may fail for denser ones.
  • Could combine with M1: bleed to full width AND scroll if even that's not enough.
Recommendation
Mobile strategy: bleed + scroll fallback
Use both M1 and M2 together:
  • Diagrams always bleed to full container width on mobile (negative margins on the diagram wrapper cancel card padding). This gets you from ~234px to ~358px usable width on a 390px screen.
  • Set a minimum readable width per diagram (480px for architecture diagrams, possibly lower for simpler flowcharts). If the container is narrower than the minimum, enable horizontal scroll.
  • Show a "scroll →" hint when scroll is active (Geist Mono 11px, ink-faint, right-aligned above the diagram).
  • On desktop (>640px), neither bleed nor scroll applies. Diagram sits within the prose column normally.
Implementation in the Astro component:
  • A <Diagram> component wraps all diagram SVGs. It handles the scroll container, bleed margins, and scroll hint.
  • Props: minWidth (default 480), caption (string).
  • CSS-only scroll detection via @container or @media query on the wrapper width.
  • No JavaScript needed. The scroll hint is always present on narrow viewports; the diagram is always scrollable if its min-width exceeds the container.
